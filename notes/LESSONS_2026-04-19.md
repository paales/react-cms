# Lessons ‚Äî eliminating the static walker

**Session:** 2026-04-19. Collapsed the remaining pieces of the
"pre-walk the JSX tree" approach into a fully runtime discovery
model, ran into the composition limit with `<Cache>`, and found the
fix that restored all previously-passing tests.

Reading order: `LESSONS_FROM_REFACTOR.md` (2026-04-18, the unified
Partial runtime) ‚Üí this doc ‚Üí `PARTIAL_ARCHITECTURE.md` (the target
state) ‚Üí `PARTIAL_CACHE_DESIGN.md` (what's next).

## 1. Removing a static walker means auditing its hidden side-effects

`seedRegistry` looked like a bootstrap helper ‚Äî "seed the route's
registry before any render runs, so first-request cache-mode
refetches can resolve." Deleting it to align with the north-star
(no static walkers) broke two things that weren't obvious from
reading the function:

- **Stale snapshot content across requests.** `seedRegistry` ran
  on *every* request, overwriting each Partial's snapshot with the
  JSX of the current request. That was the implicit mechanism
  refreshing `<Cache dep={{searchQuery}}>` when `searchQuery`
  changed. Without it, a cache-mode refetch after "p ‚Üí po" kept
  rendering stage-2 with `searchQuery: "p"` because the snapshot
  captured during the "p" request still had that value in the
  closure. `__inputs` can't reach through a `<Cache>` wrapper with
  `cloneElement` ‚Äî it only sets props on the outermost element.
- **Stale shapes surviving across refetches.** Without the seed,
  `page-2`'s registry entry from an `?end=2` render leaked into a
  subsequent `?end=3` cache-mode request, making the server think
  it could resolve page-2 from snapshot when the client's template
  didn't have a placeholder for it.

Fix: **`refreshRegistry`** (a smaller, targeted version). Same
walk, but only updates *ids already in the registry*. Doesn't add
new ids, so the registry-miss detection that powers shape-change
fallback (streaming on `page-N` when it wasn't in the last render)
still fires correctly. Paired with `clearRoute` at the start of
each streaming render ‚Äî so the two together keep registry
snapshots fresh *for ids that should exist* and absent *for ids
that no longer should*.

General principle: before deleting a function because it
contradicts the architecture, list every place it runs and what
side-effects each invocation has. Bootstrap walks and refresh
walks can live in the same function without that being obvious
from the name.

## 2. Flight composites keys; stable identity goes on a prop

A Partial produced inside `.map()` and rendered as a plain `<i>`
placeholder (not a client component) gets its own-key combined
with the outer `.map()` key by Flight:

```
// Author-side:
<Partial key="page-2" id="page-2">‚Ä¶</Partial>
  returns (from the Partial body)
<i key="page-2" hidden data-partial/>
  over Flight arrives as
key="page-2,page-2"
```

Every client-side walk that used `String(node.key)` as the id
broke. Added `data-partial-id={id}` as a prop on the placeholder
and a `getPlaceholderId(node)` helper that prefers the prop over
the key. Same shape as `partialId` on the wrapper.

Already knew this for client-component wrappers (see
`LESSONS_FROM_REFACTOR.md` ¬ß4) ‚Äî the new lesson is that plain
intrinsics like `<i>` get composited too. Anywhere the walker needs
a stable id, put it on a prop. Anywhere the walker only needs a
hint, `key` is fine.

## 3. `cloneElement` can't drill into a wrapper

`__inputs` overrides apply via `cloneElement(content, overrides)`.
That sets props on the outermost element ‚Äî which is fine for a
Partial whose content is a direct async component:

```jsx
<Partial id="stage-2">
  <SearchStage2 query={searchQuery}/>    ‚Üê override lands here
</Partial>
```

But the moment an intermediate wrapper exists, the override can't
reach through:

```jsx
<Partial id="stage-2">
  <Cache dep={{searchQuery}}>             ‚Üê override lands here (dead prop)
    <SearchStage2 query={searchQuery}/>   ‚Üê actual target ‚Äî untouched
  </Cache>
</Partial>
```

Two workarounds today:

1. The content's JSX is re-evaluated per request (via the
   `refreshRegistry` refresh), so the inner `<SearchStage2 query>`
   reads from the current closure. Works as long as the component
   reads its data from URL/context (which `<Cache>`'s `dep` does
   via closure-capture on re-render).
2. Author moves the override prop out of `<Cache>` and onto
   `<Partial>` directly (no wrapper to drill through).

The durable fix is making `<Cache>` a Partial prop (`<Partial
cache={‚Ä¶}>`) instead of a separate wrapper ‚Äî the Partial body then
owns both the override application AND the cache lookup, and it
can re-evaluate cache keys from the same props that override
propagates. Shipped as part of the auto-tracked cache key work
(see `AUTO_TRACKED_CACHE_KEYS.md`).

## 4. `startTransition` + slow Suspense children = invisibly-stale UX

Default refetch commit path in `entry.browser.tsx` is
`setPayload(payload)` which wraps the state update in
`React.startTransition`. That causes React to hold the OLD UI
visible until the new tree is ready to commit. "Ready" means no
more pending promises under the new Suspense boundaries.

Search refetches touch stage-1 (fast), stage-2 (1s), stage-3 (2s).
React holds the *whole page* on the old render until all three
resolve. Typing "p ‚Üí po" shows no feedback for ~3 seconds, then
snaps to the new results.

Pre-existing (reproduces on commits well before the 2026-04-18
refactor), but easy to mis-attribute as a regression because the
symptom is "did the refetch actually fire?" The `isPending` from
`usePartial` still flips ‚Äî it just doesn't affect any visible
node.

Two directions if this becomes a real UX concern:

1. **Default `setPayloadRaw` for `usePartial().refetch()`**. Makes
   the fallback flash on re-suspend the default, atomic-swap an
   opt-in. The per-partial `isPending` can still power smooth
   indicators on callers that want them.
2. **Per-Partial opt-out.** `<Partial suspend={false}>` bypasses
   the Suspense wrap; re-suspended children commit inline even
   under a transition. Already half-built ‚Äî the `fallback`-absent
   path skips the Suspense wrap for a different reason (see
   `LESSONS_FROM_REFACTOR.md` ¬ß2).

Not in scope for this session, but worth documenting.

## 5. Stash + checkout verify "regression vs. pre-existing"

User reported "inputting o doesn't work ‚Äî Refresh all prices
doesn't work ‚Äî LivePriceFallback doesn't show." Easy to reach for
"my recent commits broke these." Instead:

```bash
git stash                                          # hide uncommitted
git checkout 40fb36d -- src/lib src/app           # earlier commit
# run the same manual repro via dev server + playwright chromium
git checkout HEAD -- src/lib src/app              # restore
git stash pop
```

Two of the three reports reproduced at `40fb36d` ‚Äî pre-existing,
not caused by my recent work. One (the "po" snapshot staleness)
was a real regression from the `seedRegistry` removal, fixed by
`refreshRegistry`.

General principle: when a user reports "X doesn't work anymore,"
spend five minutes verifying "anymore" is accurate before
rearchitecting. The "dev server serves stale compiled code after
`git checkout`" trap is real (HMR sometimes doesn't pick up
checked-out source); run a quick manual reload after swapping
branches.

## Not-lessons worth keeping nearby

- **Dev server port drift.** Yarn dev moves to 5174/5175 when 5173
  is occupied by a stale process. All our tooling assumes 5173;
  always `lsof -nP -iTCP -sTCP:LISTEN | grep 517` when tests
  "can't reproduce" the bug you just saw.
- **`[PartialRoot] ‚Ä¶ mode=streaming registry-size=N` log format.**
  Useful template when debugging registry state over a sequence
  of requests ‚Äî paste the same `console.log` into PartialRoot's
  body, run the failing scenario, grep server-output. Remove
  before committing. (We leave these out by default; they're
  noisy in normal dev.)
- **Playwright workers: 1.** Non-negotiable given the module-level
  server caches. Parallel workers race on `<Cache>` + registry.
  Documented already in CLAUDE.md but worth repeating ‚Äî test
  failures that only happen in parallel runs are this, 95% of the
  time.
